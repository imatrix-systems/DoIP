## Stage 0: Prerequisites

### 0.1 — FC-1 HMAC secret exists and is valid
```bash
# On FC-1
ls -la /usr/qk/etc/sv/FC-1/hmac_secret
xxd /usr/qk/etc/sv/FC-1/hmac_secret | head -2
# Expect: 32 bytes, mode 0600, on yaffs2 (persistent across reboot)
```

### 0.2 — DCU HMAC secret exists and matches
```bash
# On DCU (ssh greg@192.168.7.101)
ls -la /etc/phonehome/hmac_secret
xxd /etc/phonehome/hmac_secret | head -2
# Expect: identical 32 bytes, mode 0600
# If missing: will be provisioned by FC-1 via RoutineControl 0xF0A1
```

### 0.3 — DCU SSH keypair exists
```bash
# On DCU
ls -la /etc/phonehome/id_ed25519 /etc/phonehome/id_ed25519.pub
# Expect: private key 0600, public key 0644
# Auto-generated by phonehome-keygen.sh during provisioning
```

### 0.4 — DCU phone-home config and scripts exist
```bash
# On DCU
cat /etc/phonehome/phonehome.conf
# Verify: BASTION_HOST, HMAC_SECRET_FILE, CONNECT_SCRIPT all set
ls -la /usr/sbin/phonehome-connect.sh
# Expect: exists, mode 0755
# If missing: DoIP server auto-creates all defaults on startup
```

Default phonehome.conf values:
```ini
BASTION_HOST=bastion-dev.imatrixsys.com
BASTION_PORT=22
TUNNEL_TIMEOUT=3600
HMAC_SECRET_FILE=/etc/phonehome/hmac_secret
CONNECT_SCRIPT=/usr/sbin/phonehome-connect.sh
LOCK_FILE=/var/run/phonehome.lock
```

### 0.5 — DoIP server is running on DCU
```bash
# On DCU
ps aux | grep doip-server
# If not running:
cd ~/DoIP && sudo nohup ./bin/doip-server -v > log/server.log 2>&1 &
```

Expected startup log:
```
========================================
 DoIP Blob Server v1.2.0.{build}
 Built: {date}
========================================
```

Followed by config dump and:
```
Phone-home capability enabled
```

### 0.6 — Auto-generated files (new in v1.2.0)

The DoIP server auto-creates missing files on startup:

| File | Path | Mode | Created When |
|------|------|------|--------------|
| Server config | `./doip-server.conf` (CWD) | 0644 | Before config load |
| Phone-home config | `/etc/phonehome/phonehome.conf` | 0644 | After logger init |
| Connect script | `/usr/sbin/phonehome-connect.sh` | 0755 | After logger init |
| Keygen script | `/usr/sbin/phonehome-keygen.sh` | 0755 | After logger init |
| Register script | `/usr/sbin/phonehome-register.sh` | 0755 | After logger init |

Existing files are **never overwritten**. Check server log for `Created default` messages.

## Stage 1: FC-1 Network & DoIP Connection

### 1.1 — FC-1 eth0 has an IP (LAN interface)
```bash
# On FC-1
ifconfig eth0
# Expect: inet addr on 192.168.7.x subnet
```

### 1.2 — FC-1 can ping DCU
```bash
# On FC-1
ping -c 2 192.168.7.101
```

### 1.3 — FC-1 DoIP discovery works
```
app> app:doip
```
Look for: discovered server IP, state, discovery mode.
If "no server": discovery is the problem → go to Stage 1A.

**Discovery is two-phase:**
1. **UDP broadcast** — `VehicleIdentificationRequest` to `255.255.255.255:13400`, bound to eth0 IP
2. **Unicast lease scan** (fallback) — parses `/var/lib/misc/udhcpd.eth0.leases`, probes each leased IP

Log messages:
```
[DoIP] Discovery: broadcast probe (bind=192.168.7.1)
[DoIP] Discovery result: found=1          # success
[DoIP] Discovery result: found=0          # failure, falls through to lease scan
[DoIP] Probing DHCP client 192.168.7.x    # lease scan phase
```

**Important:** Discovery gates on LAN interface (eth0 has IP), NOT on WAN connectivity.

### 1.4 — FC-1 CAN serial number is available (non-zero)
```
app> s
```
Check `CAN Bus Controller ... Serial Number:` — must be non-zero.
The CoAP URI `/remote_call_home/{can_sn}` won't register if SN=0.

## Stage 1A: Discovery Troubleshooting (if 1.3 failed)

### 1A.1 — Verify DCU is responding to UDP broadcast
```bash
# On DCU
tail -50 ~/DoIP/log/server.log | grep -i "vehicle\|discovery\|broadcast"
```

### 1A.2 — Check if UDP 13400 is open on DCU
```bash
# On DCU
netstat -ulnp | grep 13400
# Expect: udp 0 0 0.0.0.0:13400
```

### 1A.3 — Manual UDP probe from FC-1
```bash
# On FC-1 (if netcat available)
echo -ne '\x02\xfd\x00\x00\x00\x00\x00\x00' | nc -u -w3 255.255.255.255 13400
# This is a DoIP Vehicle Identification Request header
```

### 1A.4 — Check exponential backoff
Discovery retries with exponential backoff and ±25% jitter.
After multiple failures, the retry interval may be very long.
Restart the FC-1 app to reset backoff timers.

## Stage 2: CoAP Trigger Path (Bastion → FC-1)

### Understanding the Trigger Chain

```
Bastion (FastAPI)                     FC-1 Device
    │                                     │
    │  HTTP POST                          │
    │  /api/v1/devices/{sn}/phone-home    │
    ▼                                     │
CoAP Server (coap-dev.imatrixsys.com)    │
    │                                     │
    │  CoAP POST (over DTLS)              │
    │  /remote_call_home/{serial}         │
    ▼                                     ▼
    ──────────────────────────────────► CoAP handler
```

**Source:** `imatrix-bastion/web-ssh-bastion/app.py` lines 507-535

**Critical: Two different URIs on FC-1:**

| URI | Serial | Handler | Purpose |
|-----|--------|---------|---------|
| `/remote_call_home/0799874683` | FC-1 device serial | FC-1 own tunnel | Opens FC-1's own reverse SSH tunnel |
| `/remote_call_home/911934657` | CAN controller SN | DCU relay | Relays phone-home to DCU via DoIP |

The Bastion must trigger with the **CAN serial** (`911934657`) to hit the DCU relay handler.
If it triggers with the FC-1 device serial (`0799874683`), only the FC-1's own tunnel handler fires.

### 2.1 — Enable debug logging on FC-1
```
app> debug DEBUGS_APP_REMOTE_ACCESS
app> debug DEBUGS_APP_DOIP_RELAY
```

**Note:** `doip_relay:` messages are unconditionally enabled (DCU_LOG macro is always-on).
The debug flags enable additional remote access context.

### 2.2 — Verify FC-1 DTLS session is active
```
app> s
```
Look for `Cloud status: DTLS connected`.
If `DTLS in progress` or `WAIT_ONLINE`: CoAP triggers will not arrive — the DTLS tunnel
carries all CoAP traffic between Bastion and FC-1.

### 2.3 — Verify DCU CoAP URI is registered
```
app> remote
```
Should show:
```
=== DCU Relay ===
  DoIP Relay:   enabled
  HMAC Secret:  loaded
  CAN SN:       911934657
  DCU URI:      /remote_call_home/911934657
```

- URI only registers after CAN SN becomes non-zero (lazy registration)
- If SN is 0 (re-registration window), the URI is hidden — Bastion trigger has no endpoint
- If HMAC secret is missing, feature is gracefully disabled (FC-1 tunnel unaffected)

### 2.4 — Verify Bastion can reach CoAP server
```bash
# On Bastion host (or wherever web-ssh-bastion runs)
# Check configured CoAP server URL:
echo $COAP_SERVER_URL
# Default: http://localhost:3000 (or http://coap-dev.imatrixsys.com:3000 via docker-compose)

# Health check the CoAP server:
curl -s http://coap-dev.imatrixsys.com:3000/health
```

### 2.5 — Check device registration in Bastion Redis
```bash
# On Bastion host
redis-cli

# Check if device is registered (use FC-1 device serial):
HGETALL device:0799874683

# Check if CAN SN is registered as a separate device:
HGETALL device:0911934657

# Check DTLS session status:
HGETALL device:0799874683:tunnel
# Expect: status=connected
```

**Key question:** Is the CAN serial (`911934657`) registered in the Bastion's device registry?
The Bastion's `trigger_phone_home` endpoint does `get_device(r, device_id)` — if the CAN SN
isn't registered, it returns 404.

### 2.6 — Trigger DCU phone-home from Bastion

**Via Bastion API (preferred):**
```bash
# Trigger with CAN serial (for DCU relay):
curl -X POST http://<BASTION_HOST>:8000/api/v1/devices/911934657/phone-home

# Expected responses:
# {"status": "triggered", "message": "Phone home trigger sent"}      ← success
# {"status": "no_dtls_session", "message": "..."}                    ← FC-1 offline
# 404: "Device 0911934657 not registered"                            ← not in Redis
# {"status": "error", "message": "..."}                              ← CoAP server down
```

**Via CoAP server directly (bypass Bastion):**
```bash
# If CoAP server has its own trigger API:
curl -X POST http://coap-dev.imatrixsys.com:3000/api/v1/remote-access/trigger/0911934657
# This is what the Bastion calls internally (app.py line 521)
```

**Via CoAP client (lowest level — bypass everything):**
```bash
# Direct CoAP POST to FC-1 via DTLS (requires coap-client-openssl):
coap-client-openssl -m post -v 7 -B 15 \
    -u "0911934657" -k "<psk_key>" \
    "coaps://coap-dev.imatrixsys.com:5684/remote_call_home/0911934657"
```

### 2.7 — Check Bastion logs for trigger attempt
```bash
# On Bastion host — check application logs:
journalctl -u web-ssh-bastion --since "5 minutes ago" | grep -i "phone\|trigger\|911934657"

# Or if running via Docker:
docker logs web-ssh-bastion 2>&1 | tail -50 | grep -i "phone\|trigger"

# Key log lines to look for:
# "Phone home trigger failed for 0911934657: ..."  ← error in app.py line 534
# No log at all = endpoint was never called
```

**Enable debug logging on Bastion:**
```bash
# Set LOG_LEVEL=DEBUG in the bastion environment:
# Systemd: edit /opt/web-ssh-bastion/.env → LOG_LEVEL=DEBUG, then systemctl restart web-ssh-bastion
# Docker:  add LOG_LEVEL=DEBUG to docker-compose.yml environment, then docker-compose up -d
```

**In-memory log buffer (last 1000 entries):**
```bash
# If you have an active web session, retrieve via API:
curl http://<BASTION_HOST>:8000/api/v1/sessions/<session_token>/logs
```

### 2.8 — Watch FC-1 logs for trigger receipt

Expected log sequence (prefix: `doip_relay:`):
```
doip_relay: DCU phone-home triggered via CoAP
doip_relay: relay_dcu_phonehome: starting...
```

| Log Message | Meaning | Action |
|-------------|---------|--------|
| No log at all | CoAP trigger never arrived | Check 2.4-2.7 (Bastion → CoAP → FC-1 chain) |
| `rate limited` | < 60s since last relay | Wait 60s and retry |
| `HMAC not loaded` | Secret file missing/unreadable | Go to Stage 0.1 |
| `CAN SN is 0` | CAN bus hasn't provided serial | Wait for CAN registration |

### 2A — Trigger Path Troubleshooting

If no `doip_relay:` log appears on FC-1 after triggering, work backwards:

**Step 1: Did the Bastion receive the HTTP request?**
Check Bastion logs (2.7). If no log → the trigger command never reached the Bastion.

**Step 2: Did the Bastion forward to the CoAP server?**
Bastion calls `POST {COAP_SERVER_URL}/api/v1/remote-access/trigger/{sn}` (app.py line 521).
If Bastion logged an error → CoAP server is unreachable or returned an error.

**Step 3: Did the CoAP server deliver to FC-1?**
The CoAP server sends `CoAP POST /remote_call_home/{serial}` over the FC-1's DTLS session.
If CoAP server returned 404 → FC-1 has no active DTLS session, or the serial is wrong.

**Step 4: Is the serial correct?**
The Bastion formats the serial as 10-digit zero-padded decimal: `device_sn()` → `f"{device_id:010d}"`.
For CAN SN 911934657, the Bastion sends to `/remote_call_home/0911934657`.
Verify this matches the FC-1's registered DCU URI exactly (check `app> remote` output).

## Stage 3: DoIP Relay (FC-1 → DCU)

### 3.1 — Watch FC-1 logs for relay progress

Expected sequence (prefix: `doip_relay:`):
```
doip_relay: DoIP discovery...
doip_relay: DoIP discovered DCU at 192.168.7.x
doip_relay: DoIP TCP connect...
doip_relay: DoIP routing activated
doip_relay: Sending UDS RoutineControl 0xF0A0...
doip_relay: DCU response: 0x71 (positive)
```

**Relay design notes:**
- Static `doip_client_t` (avoids 4.1KB stack allocation on embedded)
- Fire-and-forget: logs result, no retry logic
- Atomic flag handoff: CoAP thread sets `dcu_phonehome_triggered`, main loop processes

### 3.2 — If discovery fails
- Check eth0 IP binding (Stage 1A)
- Check if DoIP server is listening on DCU (Stage 1A.2)

### 3.3 — If TCP connect fails
```bash
# On DCU
netstat -tlnp | grep 13400
# Verify DoIP server is listening on TCP 13400
```

### 3.4 — If UDS returns negative response (0x7F)

| NRC | Name | Meaning | Action |
|-----|------|---------|--------|
| 0x12 | SUBFN_NOT_SUPPORTED | Unknown routine ID | Routine ID mismatch — check PDU bytes [2-3] |
| 0x13 | INCORRECT_MSG_LEN | PDU too short/invalid | FC-1 building PDU wrong — check pdu_len (expect 44 bytes) |
| 0x21 | BUSY_REPEAT_REQUEST | Tunnel already active | Check DCU lock file: `cat /var/run/phonehome.lock` |
| 0x22 | CONDITIONS_NOT_CORRECT | HMAC not loaded on DCU | Check DCU config, secret file, server startup log |
| 0x24 | REQUEST_SEQUENCE_ERROR | Replayed nonce | Same nonce sent twice — check /dev/urandom on FC-1 |
| 0x31 | REQUEST_OUT_OF_RANGE | Bad data | CAN SN=0 or invalid bastion hostname chars |
| 0x35 | INVALID_KEY | HMAC mismatch | Secrets out of sync — compare hex dumps from 0.1 and 0.2 |
| 0x72 | GENERAL_PROGRAMMING_FAILURE | File write/fsync error | Check DCU disk space, filesystem |

## Stage 4: DCU Tunnel Establishment

### 4.1 — Check DCU server log for phone-home handling
```bash
# On DCU
tail -100 ~/DoIP/log/server.log | grep -i "phone\|routine\|hmac\|tunnel\|fork\|nonce"
```

Expected on success:
```
phonehome: trigger accepted. Bastion={host} Port={port} Nonce={hex}
```

Other log messages:

| Log Message | Level | Meaning |
|-------------|-------|---------|
| `phonehome: HMAC verification failed` | WARN | Possible spoofing attempt |
| `phonehome: replayed nonce detected` | WARN | Replay cache hit (64 entries, 300s TTL) |
| `phonehome: trigger rejected — tunnel already active` | INFO | Lock file held |
| `phonehome: initialized (HMAC secret loaded)` | INFO | Startup success |
| `phonehome: shutdown (HMAC secret cleared)` | INFO | Graceful shutdown |

### 4.2 — Check if connect script was spawned
```bash
# On DCU
ps aux | grep phonehome-connect
cat /var/run/phonehome.lock
# If PID exists, check if process is alive:
kill -0 $(cat /var/run/phonehome.lock) 2>&1
```

### 4.3 — If script spawned but tunnel doesn't establish
```bash
# On DCU — test SSH connectivity to Bastion directly
ssh -v -i /etc/phonehome/id_ed25519 \
    -o StrictHostKeyChecking=yes \
    -o UserKnownHostsFile=/etc/phonehome/known_hosts \
    -o ConnectTimeout=10 \
    phonehome-$(cat /etc/dcu-serial)@<BASTION_HOST> \
    echo "test"
# Watch for: key rejected, host key mismatch, connection refused, timeout
```

Connect script SSH parameters:
```
timeout 3600   (1 hour tunnel lifetime)
-R {remote_port}:localhost:22222   (reverse tunnel to local SSH)
ServerAliveInterval=30, ServerAliveCountMax=3
StrictHostKeyChecking=yes, BatchMode=yes
ExitOnForwardFailure=yes
```

### 4.4 — Check known_hosts
```bash
# On DCU
cat /etc/phonehome/known_hosts
# If empty/missing: DCU can't verify Bastion identity
# SSH will fail with StrictHostKeyChecking=yes
```

### 4.5 — Check Bastion accepts DCU's public key
```bash
# On Bastion
grep "$(cat /etc/phonehome/id_ed25519.pub | awk '{print $2}')" ~/.ssh/authorized_keys
# Or check Bastion's device registry for DCU's serial
```

## Build & Deploy

### Build DoIP Server
```bash
cd ~/iMatrix/DOIP/DoIP_Server
make clean && make
# Version auto-increments BUILD_NUMBER on each build
# Output: ./doip-server
```

### Deploy DoIP Server to DCU
```bash
scp ~/iMatrix/DOIP/DoIP_Server/doip-server greg@192.168.7.101:~/DoIP/bin/doip-server
# Then on DCU: restart the server
```

### Build & Deploy FC-1
```bash
cd ~/iMatrix/DOIP/Fleet-Connect-1
mkdir -p build && cd build && cmake .. && make -j$(nproc)
# Deploy:
cd ~/iMatrix/iMatrix_Client && ./scripts/fc1 -d 192.168.7.1 -b ./DOIP/Fleet-Connect-1/build/FC-1 push -run
```

**Note:** The DOIP worktree build path is `~/iMatrix/DOIP/Fleet-Connect-1/build/FC-1`, NOT `~/iMatrix/iMatrix_Client/Fleet-Connect-1/build/FC-1`. Using the wrong path deploys a stale binary.
